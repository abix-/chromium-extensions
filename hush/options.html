Hush

Firewall-style rule engine for Chrome. Seven actions, per-site or global scope, behavioral detector that suggests rules from what the page actually does.

How Hush works

Hush is a firewall for your browser. Every request a page makes, every element it renders, and every fingerprint API it reads is checked against rules you authored. Matched things get a chosen action: dropped, allowed, neutered, silenced, removed, hidden, or spoofed. All logic runs locally; Hush itself never sends anything to any server.

A rule is a (scope, action, match) triple. Scope is either a hostname or the reserved Global key. Action is one of the seven below. Match is a URL pattern, a CSS selector, a script-origin host pattern, or a fingerprint kind tag - depending on the action. Rules are evaluated first-match-wins within each action.

The seven actions

1Block (network) — kills the request before it leaves the browser

Patterns in the Block list register rules in Chrome's declarativeNetRequest engine. A matching request is refused before DNS resolution, TCP connection, or TLS handshake. No bytes go to any server. The originating fetch(), <script> load, or iframe load fails locally with net::ERR_BLOCKED_BY_CLIENT.

Why it's relevant. A modern news or social page loads 40-200 third-party requests on first paint. Every one of those is a potential tracker, telemetry pipe, or fingerprint probe. Block is the most complete defense available to an extension: no script runs, no response is parsed, no iframe loads. If you can identify the URL, Block wins. Chrome's DNR enforces at the network layer with zero extension-JS involvement, so it has no perf cost on matched requests.

Rules are declared under a site in your config but applied as global URL-pattern matches. A rule under reddit.com blocks its target URL anywhere it's requested - including from third-party iframes embedded on Reddit. Site-keyed authoring is how you organize your rules; execution is global by URL.

Pattern syntax (uBlock-style):

||domain.com match requests to domain.com and any subdomain, across any path/port ||ads.example.com match requests to exactly ads.example.com or its subdomains ||tracker.net/pixel match requests whose path starts with /pixel *.cdn.example.com wildcard on subdomain (* = any characters) ^ separator anchor. Rarely needed after ||domain; can be problematic with hyphenated subdomains.

When to pick Block: the request itself is what you want gone - tracking pixels, ad scripts, telemetry beacons, widget iframes, API calls to unwanted services.

2Allow (exception) — the counter-rule to Block

Patterns in the Allow list override Block rules via DNR priority. For Remove/Hide, an Allow selector excludes matching elements from those passes.

Why it's relevant. Sometimes you want to Block a domain globally but carve out a single site-specific path. Example: global Block on ||doubleclick.net; Allow ||doubleclick.net/specific-study/ only on research.example.com. Without Allow you'd have to delete the Block; with Allow you keep the default protection and poke one hole.

Pattern syntax: uBlock-style URL patterns for network Allow, CSS selectors for DOM Allow.

When to pick Allow: whenever a global rule is too broad for one specific site.

3Neuter (capture denial) — stops session-replay and engagement trackers at registration

At document_start, Hush wraps EventTarget.prototype.addEventListener. When a script whose origin matches your Neuter pattern tries to register an interaction listener (mousemove, keydown, click, scroll, wheel, touch*) or a lifecycle listener (visibilitychange, focus, blur, pagehide, pageshow, beforeunload), the registration is silently dropped.

Why it's relevant. Session-replay vendors (Hotjar, FullStory, Clarity, LogRocket, Smartlook, Mouseflow) record every mouse movement, scroll, keystroke, and page-visibility change and ship the trace to their servers for later playback. Curated blocklists can only kill these when the vendor's script loads from a known CDN; when the site ships a bundled replay library (increasingly common), the list can't help. Neuter intervenes one layer upstream: no listener registered = no capture possible, regardless of where the script came from.

Pattern syntax: host-anchor pattern against the initiating script's host, not the page URL.

||hotjar.comany script loaded from hotjar.com or a subdomain ||*.fullstory.comany fullstory.com subdomain ||vendor-cdn.example.com^trailing ^ accepted as a no-op

When to pick Neuter vs Block: use Block when you can kill the vendor's script URL (upstream-most defense). Use Neuter when you can only identify the vendor by the origin of calls into the listener API - typically when the script is bundled into first-party code and has no distinct network identity.

4Silence (exfil denial) — fake-succeeds outbound telemetry

Hush wraps window.fetch, XMLHttpRequest.send, navigator.sendBeacon, and WebSocket.send. When the calling script's origin matches your Silence pattern, the call returns a fake success (HTTP 204 for fetch/XHR, true for sendBeacon) without actually writing to the network. The calling code sees no error.

Why it's relevant. Fallback for capture libraries that Neuter can't reach - the ones that read directly from event streams or DOM state without registering listeners through the public API. If you can't stop them from collecting, stop them from exfiltrating.

Pattern syntax: same host-anchor pattern grammar as Neuter, matched against the initiating script's host.

When to pick Silence: when Neuter can't match by script origin (bundled library, anonymous source-map obfuscation) but you can still attribute the exfil calls to a specific script host.

5Remove (DOM) — physically deletes elements

Selectors in Remove are deleted via element.remove(). A MutationObserver watches every DOM change; elements inserted later by SPA routers, infinite-scroll, modal mounts, or ad auctions are removed as soon as they appear.

Why it's relevant. Promoted posts, algorithmic feed recommendations, cookie banners, newsletter popups, sticky headers, "Brand Affiliate" tags, sidebar widgets - none of these are on public blocklists because they're site-specific custom elements. Remove takes them out permanently. Remove also frees event listeners and stops animations tied to the removed element, which often cuts CPU burn on idle tabs.

Pattern syntax: any valid CSS selector.

.modal-overlay
[data-role="promo"]
iframe[src*="tracker.example.com"]
article[data-post-id]:has([is-post-commercial-communication])

When to pick Remove vs Hide: Remove for listeners/animations/JS-dependencies to stop. Hide when the site's own JS crashes if its DOM gets mutated (rare but real).

6Hide (CSS) — element stays in the DOM but doesn't render

Selectors in Hide are applied as a user stylesheet with display: none !important. The browser skips layout, paint, compositing, and GPU work for hidden elements; CSS animations on those elements also stop.

Why it's relevant. The mildest layer. Useful when Remove would break the page (some SPAs crash when their own DOM is mutated from under them), or as a safety net for elements that might exist after you've blocked their network resources at the Block layer.

Pattern syntax: any valid CSS selector (same as Remove).

When to pick Hide: visual annoyances where Remove risks breaking the page. Also useful as a cross-pass fallback: the Block layer kills the request, Hide cleans up the empty shell the site's JS leaves in the DOM.

7Spoof (fingerprint) — returns bland constants from fingerprint APIs

Hush intercepts specific fingerprinting APIs and returns identical-across-users values instead of real ones. The calling site still gets a response, it just can't identify you by it.

Why it's relevant. Browser fingerprinting combines 3-4 signals (GPU model + canvas rendering + installed fonts + audio DSP quirks) to identify 90%+ of browser sessions regardless of cookies, incognito mode, or VPN. Because fingerprinting happens at read time - not a separate request - Block can't stop it, and the page would break if you simply denied the read. Spoof lets the page keep rendering while killing the entropy.

Fingerprint kinds: return bland constants to kill entropy contribution without breaking the page.

webgl-unmasked getParameter(UNMASKED_VENDOR_WEBGL) / UNMASKED_RENDERER_WEBGL return "Google Inc." / "ANGLE (Generic)" instead of the real GPU model. canvas toDataURL / toBlob return a constant 1x1 PNG; getImageData returns zero bytes. Kills subpixel-rendering fingerprints. Opt-in per site because legit canvas use breaks. audio OfflineAudioContext.startRendering resolves to a silent buffer. Kills audio-DSP-divergence fingerprints. font-enum measureText returns length-only metrics. Kills installed-font probing via width measurement.

Kill-switch kinds: for APIs that have no legitimate user-side benefit. When any of these is set in the Global scope, the API is disabled site-wide. Much simpler than authoring per-site Block rules, and defaults-on for new installs (see sites.json) so you get the defense without configuring anything.

sendbeacon navigator.sendBeacon returns true without firing. The API's literal purpose is page-unload telemetry - there's no user-side beneficiary. Kills session-end analytics, dwell-time reports, and "did-the-user-close-the-tab" beacons. Zero known collateral damage. clipboard-read navigator.clipboard.readText rejects with NotAllowedError - the exact DOMException the browser raises when the user declines the permission prompt. Kills coupon-sniffing and paste-in analytics. Legit page-script use is near-zero (password managers and clipboard tools run as extensions). bluetooth Bluetooth.requestDevice rejects with NotFoundError. Remove from your config if you actively use WebBluetooth (maker-space dashboards, physical-web integrations). usb USB.requestDevice rejects. Remove if you use WebUSB. hid HID.requestDevice rejects. Remove if you use WebHID (gamepads, specialized peripherals). serial Serial.requestPort rejects. Remove if you use WebSerial (Arduino IDE web, flashing tools).

Why kill-switches are Global-scope Spoof rules: the existing Spoof primitive already matches what we want - intercept the API, return a spec-compliant "not happening" value, let the page handle the denial. Global scope applies it everywhere. New kinds are a one-line addition; no new schema, no new UI, no migration.

When to pick Spoof vs Block: Block when the fingerprint script has a distinct URL you can target (upstream-most). Spoof when the read happens in bundled first-party code or in an API call that has no URL to block against (sendBeacon, clipboard, device probes).

Scopes: site-specific vs Global

Where rules evaluate

Every rule has a scope. A site-scoped rule evaluates only on tabs whose hostname matches the scope key (exact or suffix - reddit.com also covers www.reddit.com, old.reddit.com, sh.reddit.com). The reserved Global scope (__global__ under the hood) evaluates on every tab.

Why it's relevant. Global is for rules that should apply everywhere: known-bad ad networks, session-replay vendors you never want firing, fingerprint scripts from anywhere. Site-specific is for surgical per-site cleanup: Reddit's promoted posts, GitHub's telemetry collector, Amazon's ad iframes. Without Global you'd duplicate the same rule across every site entry; without site-specific you'd over-block.

Pattern: put broad vendor bans in Global, put page-structure cleanup in the site scope. The two layers combine: when a rule exists in both, the site scope wins dedup.

The behavioral detector (opt-in)

What it watches, what it suggests, why the curated lists can't do this

Off by default because it costs some per-scan CPU. Once on, Hush observes what the current page actually does and surfaces rule suggestions in the popup. Every suggestion carries raw evidence (URLs, sizes, stack traces, outerHTML snippets) so you verify before accepting. Every scan runs locally; no lists fetched, no observations ever leave the current tab.

Why this exists. Public blocklists (EasyList, EasyPrivacy, uBO, AdGuard, Privacy Badger) only know about publicly-documented trackers. First-party telemetry subdomains, site-specific custom elements, session-replay listener patterns, and fingerprint-API reads don't generalize across millions of users, so no curated list captures them. An observe-and-suggest engine closes exactly that gap.

Network-layer signals (from PerformanceObserver):

  • sendBeacon targets (confidence 95) - navigator.sendBeacon is purpose-built for fire-and-forget telemetry that survives page unload. Any third-party host receiving beacons is almost always a tracker.
  • Tracking pixels (85) - third-party <img> responses under 200 bytes. The classic 1x1 pixel-tracker pattern.
  • First-party telemetry subdomains (70) - subdomains of the current site whose responses are all tiny (median under 1 KB). GitHub's collector.github.com, Amazon's unagi.amazon.com, Reddit's w3-reporting.reddit.com are textbook examples. No curated list blocks these by policy.
  • Polling endpoints (75) - the same canonical URL fetched 4+ times in seconds, with tiny responses. Heartbeats, long-poll loops, telemetry rollups.
  • Hidden iframes (80, with default allowlist for captcha/OAuth/payment) - iframes rendered with display:none, visibility:hidden, 1x1 size, or offscreen. Legitimate uses are captcha / OAuth / payment, all allowlisted by default; the rest are ad containers or tracking pixels.
  • Sticky overlays (55) - fixed-position elements with z-index >= 100 covering >= 25% of viewport. Typically nag popups, cookie banners, subscribe prompts.

Fingerprint signals (from main-world API hooks):

  • Canvas fingerprinting (90) - 3+ calls to toDataURL / toBlob / getImageData from one script origin. Canvas rendering is the most-used browser-fingerprint signal; a few reads per page is diagnostic.
  • WebGL fingerprinting (95 for UNMASKED_* reads, 75 for general getParameter flurry) - reading the GPU vendor/model strings is fingerprint-specific; no legitimate use exists for identifying the exact GPU.
  • Audio fingerprinting (90) - any OfflineAudioContext construction. The context exists almost exclusively for DSP-divergence fingerprinting.
  • Font enumeration (85) - measureText across 20+ distinct font families in one session. Installed-font probing via width measurement.
  • Navigator / screen accessor density (Tier 3) (80) - 10+ distinct reads across navigator.userAgent, platform, languages, hardwareConcurrency, deviceMemory, maxTouchPoints, etc. within 60s. Layout-dependent properties (innerWidth, devicePixelRatio) deliberately excluded so the signal stays fingerprint-specific.

Session-replay signals:

  • Vendor globals (95) - Hush polls for window._hjSettings (Hotjar), window.FS (FullStory), window.clarity (Microsoft Clarity), window.LogRocket, window.smartlook, window.mouseflow, window.__posthog. Presence of any is diagnostic.
  • Listener density (80) - 12+ mouse/keyboard/scroll listeners attached to document/window/body from one script origin within 60s. Suggests Neuter rather than URL block: it's the listener registration itself that's the fingerprint, not the vendor domain.

Attention / lifecycle tracking:

  • Visibility / focus / pagehide density (75) - 4+ listeners across visibilitychange, focus, blur, pagehide, pageshow, beforeunload from one origin. Catches engagement analytics and session-replay dwell-time hooks. Brave Shields doesn't specifically target this; Hush does. Suggests Neuter.

API-probe signals:

  • Clipboard read (95) - any call to navigator.clipboard.readText() from a page script. Chrome gesture-gates the API, but legitimate page-script use is near-zero (password managers run as extensions). Catches coupon sniffing, competitor-URL pasting, paste-in analytics.
  • Hardware device probes (90) - calls to Bluetooth.requestDevice, USB.requestDevice, HID.requestDevice, Serial.requestPort. Legitimate uses are maker-space / industrial / dev-tool contexts. Random web pages calling these are fingerprint probes - the permission prompt itself is an entropy signal. Brave doesn't hook any of these.

Performance-abuse signals:

  • Invisible animation loop (70) - 20+ canvas draw ops over >=3s against a canvas that's >=80% invisible (display:none, offscreen, sub-2px). Catches Lottie-style animations running inside collapsed panels, hidden widgets, or auto-hidden ads still burning CPU.

How suggestions appear: the toolbar icon shows a yellow ! badge on tabs with pending suggestions. Click to review. Each row has three buttons:

  • + Add - writes the rule. A scope picker asks: site-specific or global?
  • Dismiss - suppresses this suggestion for this tab session only. Reloading brings it back.
  • Allow - permanent across every site, saved to the suggestion allowlist. Revocable from the options allowlist editor below.

Manual override: even with the feature off, the popup's Scan this tab now button runs a single scan on demand.

The firewall log (popup)

Every rule hit, aggregated and filterable

Every time a rule fires, Hush records one event with: timestamp, action, scope, match, and action-specific evidence (blocked URL + resource type for Block/Allow, element description for Remove, kind tag for Spoof). Events accumulate in a global ring buffer capped at 10000, persisted to chrome.storage.session (survives popup close and tab reload; clears on browser restart).

Why it's relevant. "Did my rule actually fire?" is the number-one question when tuning. The firewall log answers it with receipts. Rules that never fire are dead code; rules that fire 10k times are probably too broad. The popup aggregates by rule ID and shows hit count, last-hit time, and recent evidence - you can spot broken or shadowed rules at a glance.

Health indicators shown per row:

  • shadowed by allow - an Allow rule covers this Block; it will never fire.
  • no DOM match on this tab - a Remove/Hide selector that matched nothing since the last navigation. Could be a typo, could be legitimate (the element didn't appear on this page).
  • invalid selector - the CSS parser rejected the selector. Usually a missing quote or unescaped metachar.

Filters: tab (this tab / all tabs), action, tags, substring search over match.

Rule simulate (test-match)

Check what rules fire for a given URL before saving

The options page has a Simulate panel. Paste a URL and optionally a site hostname; Hush walks your config and returns every matching rule across the Global scope and the site scope, with action / value / priority. The DNR winner is flagged with a check.

Why it's relevant. You don't want to find out a rule is broken by clicking around the actual site. Simulate is a dry run against your current config: if a pattern should match and doesn't, you see that before saving. Shares code with the real evaluator so simulation can't diverge from production behavior.

Rule profiles (import/export)

Ship entire rule sets between machines or between users

The options page has Import Profile and Export as Profile buttons. Export wraps the current config with a user-supplied name + description inside a hushProfile header and triggers a JSON download. Import parses profile JSON, previews the merge (added vs skipped rule counts), then unions the profile's rules into live config. Existing rule metadata (disabled, tags, comment) is preserved; new rules append to their bucket.

Why it's relevant. Your carefully-tuned Reddit / GitHub / Amazon rules are portable. Share a profile with someone else and they get your rule set without typing a single selector. Curated starter profiles ship alongside Hush for common scenarios (news-site baseline, social-media declutter, Brave-stack supplement).

Allowlists

What the detector should skip (NOT what rules are exempt)

Three detector-scoped allowlists, all user-editable in the Advanced section below:

  • iframes - URL substrings. A hidden iframe whose src matches any entry is skipped by the hidden-iframe detector. Defaults cover captcha, OAuth, payment widgets.
  • overlays - CSS selectors. A sticky overlay matching any selector is skipped by the sticky-overlay detector. Defaults cover React Portals and framework modal containers.
  • suggestions - full suggestion keys. Populated by the popup's Allow button. Once in this list, the suggestion never resurfaces on any site.

Why it's relevant. Allowlists suppress suggestions, not rules. If you want a stored rule to stop firing, delete the rule. If you want a specific suggestion to stop appearing, Allow it. The two concerns are separate, so the primitives are separate.

Cross-layer interaction

How the seven actions combine when they match the same thing
  • Allow over Block: DNR's priority resolution gives Allow rules higher priority than Block. For DOM layers, Allow selectors exclude nodes from Remove/Hide passes.
  • Remove over Hide: Remove runs first on every MutationObserver pass, so Hide sees an empty selector set. The popup shows such selectors as - (removed) in the Hide section; that's honest reporting, not a bug.
  • Block plus Hide: Block kills the request; if the site's JS still builds a DOM element with a failed src, Hide suppresses the empty shell.
  • Neuter over Silence: Neuter blocks the listener from registering; Silence intercepts the exfil call after registration. Use Neuter when you can match by script origin; Silence as the fallback.
  • Block rules are global URL patterns. A rule under any site scope blocks that URL everywhere - including in embedded iframes on other sites. Site-keyed authoring is organizational, not a runtime scope. For a URL blocked on one site only, narrow the pattern itself.
  • Iframe traffic is attributed to the tab. A request from an embedded ad iframe on Reddit counts as Reddit-tab activity. Your Reddit rules cover it automatically.

Runtime order for any given element:

  1. Block / Allow decide whether the request happens at all.
  2. Neuter decides whether capture listeners ever register.
  3. Silence intercepts any capture call that still made it out.
  4. Remove decides whether the element stays in the DOM.
  5. Hide decides whether what remains actually renders.
  6. Spoof returns bland values from any fingerprint reads.

Data egress

Where your config and observations live

Nothing leaves the extension. Hush does not fetch filter lists, does not send observations anywhere, does not phone home. All analysis runs locally in the current tab or the service worker.

Your config (chrome.storage.local) contains every rule you authored; it syncs across Chrome profiles only if you explicitly enable sync. The firewall log lives in chrome.storage.session and clears on browser restart. Behavioral observations are per-tab and wiped on navigation.

Allowlists (things the behavioral detector ignores)

When the behavioral-suggestions detector would otherwise flag these as suspicious, it skips them instead. Default entries cover captcha, OAuth, payment iframes, and framework portal containers. Add your own site-specific entries as you encounter false positives.
